# ALERT — API 500 on launch day

**Timestamp (UTC):** 2026-04-25 12:53:10
**Severity:** High — public API endpoint failing on launch day

## What's failing
- **Endpoint:** `GET https://api.world-pulse.io/api/v1/public/signals?limit=1`
- **HTTP status:** 500 Internal Server Error
- **Postgres error code:** 42703 (undefined_column)

## Error body
```json
{"statusCode":500,"code":"42703","error":"Internal Server Error",
 "message":"select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"}
```

## Diagnosis
The public signals route is selecting `source_url` directly from the `signals` table, but that column does not exist in the prod schema. Per memory (`reference_api_slop_routes.md` / `project_signals_schema_drift.md`) the canonical column is `source_ids` (text[]) which joins to `sources`. This is a schema-drift / query-mismatch bug, not a DB outage — the same failure has been logged repeatedly today (21:36, 18:36, 20:19 UTC entries all show API: FAIL with N/A signals).

## Other checks
- Web (https://world-pulse.io): **200 OK**
- GitHub stars (World-Pulse/WorldPulse): **3**

## Recommended fix
1. Patch the public signals route to either:
   - drop `source_url` from the SELECT, or
   - LEFT JOIN `sources` on `source_ids` and project `sources.url AS source_url` (cast both sides ::text per `project_signal_detail_500.md`).
2. Redeploy API (`docker compose -f docker-compose.prod.yml up -d --force-recreate api` — restart alone won't pick up code changes per memory).
3. Re-run this health pulse to confirm signals count returns.
