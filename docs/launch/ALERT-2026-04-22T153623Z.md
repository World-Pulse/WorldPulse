# ALERT — Public API 500 (persistent schema drift)

**Timestamp:** 2026-04-22 11:35 local / 15:36:23 UTC
**Severity:** HIGH (public /v1/public/signals returning 500)
**Status:** ONGOING — same failure observed across prior polls on Apr 19, 20, 21, and 22

## What failed
- `GET https://api.world-pulse.io/api/v1/public/signals?limit=1` → **HTTP 500**
- Response body:
  ```json
  {"statusCode":500,"code":"42703","error":"Internal Server Error",
   "message":"select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"}
  ```
- Web: `https://world-pulse.io` → HTTP 200 (OK)
- GitHub stars: 3 (unchanged)

## Root cause (already diagnosed in prior alerts)
Schema drift: `signals` table has no `source_url` column; public signals route selects it directly. Source URLs live in the `sources` table and are resolved via `source_ids text[]` join (per `reference_api_slop_routes.md` and `project_signal_detail_500.md` memories).

## Required fix
Patch the public signals query in the API (`apps/api/src/routes/public/signals.ts` or equivalent) to either:
1. Drop `source_url` from the SELECT and resolve primary source URL via `LEFT JOIN sources s ON s.id = ANY(signals.source_ids::text[])` and alias `s.url AS source_url`, OR
2. Add a generated/materialized `source_url` column on `signals` populated from `sources.url` of the first `source_ids` entry.

Option 1 is cheaper and matches the pattern already used elsewhere.

## Impact
- Public signals feed endpoint has been down continuously since at least Apr 19. Homepage ingest count still climbing (70,336 → 73,772 last successful scrape), so internal pipeline is healthy — only the public read path is broken.
- SDK/embed/API consumers cannot read signals via the documented public route.
- Launch narrative ("open-source intelligence API") is undermined while this is 500.
