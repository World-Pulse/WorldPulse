## Likely culprits (from memory)

- `wp_api` container: known to build with `pnpm build || true` (100+ TS errors shipped). A runtime crash-loop would present as 500 at the edge.
- nginx upstream config must use `wp_api` (container name), not `api` (compose service) — check `nginx.conf`.
- "Container Started" ≠ healthy — a crash-looping container logs "Started" every cycle. Verify with `docker inspect` + direct curl to `wp_api:3001` inside the network.
- Recent env-var changes require `docker compose up -d --force-recreate`, not just `restart`.

## Recommended next steps (human action required)

1. SSH to prod. `docker ps` and check `wp_api` status + restart count.
2. `docker logs wp_api --tail 200` — look for crash stack / startup error.
3. From inside the compose network: `docker exec wp_nginx curl -sS http://wp_api:3001/healthz` — isolates nginx→api vs. edge→nginx.
4. If api crash-looping, check `.env.prod` for missing/malformed vars since last deploy.
5. Once resolved, run this scheduled task on-demand to confirm recovery.
