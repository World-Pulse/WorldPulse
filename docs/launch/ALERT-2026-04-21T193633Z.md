## Likely suspects (from prior memory, verify before acting)

1. **wp_scraper / wp_api container crash-looped** — `wp_scraper` is the single point of failure for ingest+classify+correlate+trending per `project_scraper_pipeline.md`. If it's down, the API may be returning 500 from downstream timeouts rather than a missing route.
2. **OpenSanctions rate-limit blowback** — free-tier ~60 req/hour cap (see `project_opensanctions_rate_limits.md`); check `/sanctions/featured` is still using the seed fallback and not hot-calling the live API on every `/signals` request.
3. **DB/schema drift or enum mismatch** — prior Apr 15 fixes around `alert_tier`, `source_ids::text`, and the `category` enum (see `project_apr15_prod_fixes.md`, `project_signal_detail_500.md`). A fresh migration or a rollback could have re-introduced drift.
4. **Container "Started" ≠ "healthy"** — per `feedback_container_started_vs_healthy.md`, verify with `docker inspect` + `curl` + `logs`, not just `docker ps`.

## Recommended next actions (for a human on-call)

1. SSH to prod, run on the server:
   - `docker compose -f docker-compose.prod.yml ps`
   - `docker compose -f docker-compose.prod.yml logs --tail=200 wp_api wp_scraper`
