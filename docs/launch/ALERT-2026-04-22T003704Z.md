## Suspected causes (from memory)

- `wp_scraper` is a single point of failure for ingest/classify/correlate/trending; if scraper is dead the API's DB may still return but `signals` query could be erroring on a downstream dependency (e.g., stale materialized view, redis dependency, or a query that joins on empty data).
- Per `project_signal_detail_500.md`: `signals.source_ids` is `text[]` in prod — any recent join with `sources.id` without `::text` cast throws 500.
- Per `feedback_restart_vs_recreate.md`: if env vars were changed, `docker compose restart` won't re-read them — need `up -d --force-recreate`.

2. Verify `wp_scraper` is running and healthy (`scraper:health:*` HGETALL in redis, password `wp_redis_local`).
