## Suspected causes (from memory)
- `wp_scraper` is a single point of failure for ingest/classify/correlate/trending — if the signals API depends on scraper-populated caches or tables, a crashed scraper could cascade.
- Recent schema-drift fixes (Apr 15) addressed signals.source_ids text[] cast issues; a regression in the public signals query path would reproduce as 500.
- API Dockerfile uses `pnpm build || true` and ships with 100+ TS errors — a runtime path unhandled until hit.
- Check `/api/v1/slop` vs `/api/v1/public/signals` — slop endpoints might still be live even if public signals handler is crashing.

## Recommended immediate actions for Devon
1. SSH to prod. `docker compose -f docker-compose.prod.yml ps` — confirm wp_api + wp_scraper states; wp_api showing "Started" is NOT the same as healthy (see feedback memory).
2. `docker compose -f docker-compose.prod.yml logs --tail=200 wp_api` — find the 500 stack trace on `/api/v1/public/signals`.
3. `curl -sS http://localhost:3001/api/v1/public/signals?limit=1 -w "\n%{http_code}\n"` from the host to bypass nginx and confirm the 500 is from the API container itself.
4. If stack trace points at enum/cast drift again, compare prod schema vs migrations in repo (signal category enum is named `category`, not `signal_category`).
5. If scraper is crash-looping, restart scraper first — many API read paths depend on its materialized state.
