```json
{"success":false,"code":"VALIDATION_ERROR","error":"offset: Expected string, received number"}
```

Today's probe returned 500 without a body visible to the monitor. Two possibilities:

1. The origin has regressed from a 400 to a 5xx (worse — the process or upstream is now erroring instead of cleanly rejecting).
2. The WebFetch gateway surfaced a generic 500 while the origin still returns 400.

Either case warrants a direct `curl` from a human operator to disambiguate.

Four alerts filed on 2026-04-19 (21:25Z, 21:36Z, 22:06Z, 22:36Z) documented the original 400 VALIDATION_ERROR on `offset`. Recommended patch was to change the route's zod schema to `z.coerce.string()` / union / default `"0"`, or disable upstream numeric coercion for the route's querystring. The patch has not landed — every subsequent probe through 15:05 today still fails.
