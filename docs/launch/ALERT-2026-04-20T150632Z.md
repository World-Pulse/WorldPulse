# ALERT — API 500 on Launch Day (schema drift identified)

**Time:** 2026-04-20 11:06 local (15:06:32Z)
**Severity:** HIGH — launch day, public API endpoint down ~18h
**Status:** Same standing incident as ALERT-2026-04-19T212516Z and later. NEW diagnostic info below.

## What failed

- `GET https://api.world-pulse.io/api/v1/public/signals?limit=1` → HTTP 500
- Web (`https://world-pulse.io`) → HTTP 200, OK
- GitHub repo reachable, 2 stars

## NEW — Root cause surfaced via direct curl

Previous alerts could not parse the 500 body (WebFetch drops error payloads). A direct `curl` this run returned the actual error:

```json
{
  "statusCode": 500,
  "code": "42703",
  "error": "Internal Server Error",
  "message": "select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"
}
```

Postgres error code `42703` = undefined column. The API is selecting `signals.source_url`, but that column does not exist on the prod `signals` table. This is schema drift — code expects a column the database doesn't have.

## Why this matters

1. No amount of container-restart will fix this. It is NOT a crash loop; the API is up and serving — it's returning 500 because the query references a missing column.
2. This is almost certainly related to the known TypeScript-debt shipping pattern (`pnpm build || true` in `project_api_typescript_debt.md`) plus the signals schema history in `project_signals_schema_drift.md`. A handler was shipped that references a column that was either renamed, dropped, or never migrated.
3. Public API is the primary surface for the `/api/v1/public/signals` endpoint used by the live homepage feed. The web still returns 200 because the homepage can fall back to cached/seed data, but any fresh signal query fails.

## Recommended next actions (in order)

1. **Identify the column reality.** SSH prod → `psql $DATABASE_URL -c "\d signals"` and verify whether `source_url` exists. Likely candidates it was renamed to: `source_urls`, or data lives in `source_ids` (text[]) and joins `sources.url` (`project_signal_detail_500.md` covered a similar casting issue).
2. **Find the offending handler.** In the `api` package, grep for `source_url` in the signals route/query. The SQL suggests it's a simple select against `signals` with a `status` filter and `order by created_at desc` — likely the public signals feed handler.
3. **Decide: fix query or add column.** If `source_url` was intentionally dropped in favor of `source_ids[]`, fix the SELECT to either drop the field or join `sources` and return a url. If the column should exist, run a migration to add it.
4. **Deploy.** Use `deploy-bg.ps1` (per `project_deploy_pipeline.md`), and recreate with `--force-recreate` (per `feedback_restart_vs_recreate.md`). Wait for healthy before running any follow-up deploy (per `feedback_deploy_bg_concurrency.md`).
5. **Smoke test.** `curl https://api.world-pulse.io/api/v1/public/signals?limit=1` should return 200 with a JSON array of one signal.
