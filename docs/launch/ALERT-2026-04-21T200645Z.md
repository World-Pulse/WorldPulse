## Likely suspects (from prior memory; verify on-host, don't act blind)

1. **wp_scraper / wp_api container crash-loop** — `wp_scraper` is the single point of failure for ingest+classify+correlate+trending (memory: `project_scraper_pipeline.md`). The API likely returns 500 from downstream timeout rather than a missing route.
2. **Container "Started" ≠ "healthy"** — check with `docker inspect --format '{{.State.Health.Status}}'`, `curl localhost:3001/api/v1/public/signals?limit=1` on-host, and tail `docker logs wp_api --tail 200` (memory: `feedback_container_started_vs_healthy.md`).
3. **OpenSanctions rate-limit blowback** — free-tier ~60 req/hour cap; verify `/sanctions/featured` is still hitting `sanctions-seed.ts` fallback, not hot-calling the live API on every `/signals` request (memory: `project_opensanctions_rate_limits.md`).
4. **DB/schema drift regression** — Apr 15 fixes around `alert_tier`, `source_ids::text`, `category` enum may have been re-introduced by a later deploy or rollback (memory: `project_apr15_prod_fixes.md`, `project_signal_detail_500.md`).
5. **Env var not reloaded** — if someone did `docker compose restart` instead of `up -d --force-recreate`, env vars from a recent `.env.prod` edit are not actually in the container (memory: `feedback_restart_vs_recreate.md`).
