## Likely suspects (from memory)

- `wp_scraper` single-point-of-failure for ingest/classify/correlate/trending — if it crash-looped, downstream signal queries could 500 on empty/broken joins. (See `project_scraper_pipeline.md`.)
- `signals.source_ids` text[] cast mismatch — previously caused 500s on signal detail; worth checking if it regressed to the list endpoint. (See `project_signal_detail_500.md`.)
- Container "Started" ≠ healthy — `docker compose ps` can lie. Verify with `docker inspect` + `curl` against the in-container healthcheck + logs. (See `feedback_container_started_vs_healthy.md`.)
- If a recent env change: `docker compose restart` does NOT re-read env — need `up -d --force-recreate`. (See `feedback_restart_vs_recreate.md`.)

## Suggested triage (for whoever picks this up)

1. SSH to prod, run `docker compose -f docker-compose.prod.yml ps` and `docker inspect wp_api --format '{{.State.Health.Status}}'`.
2. `docker compose -f docker-compose.prod.yml logs --tail=200 wp_api wp_scraper` — look for a stack trace or repeated error.
3. Hit the API locally on the box: `curl -s http://localhost:3001/api/v1/public/signals?limit=1 -w '\n%{http_code}\n'` — confirms whether it's the API container itself or the edge (nginx/Cloudflare) in front of it.
4. If scraper is crash-looping, tail its logs for the first error after restart — don't trust the repeating line.
