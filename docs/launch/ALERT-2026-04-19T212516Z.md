# ALERT — Public signals endpoint returning HTTP 400

**Timestamp (UTC):** 2026-04-19T21:25:16Z
**Check time (local):** 17:24
**Severity:** HIGH — launch day, primary public data endpoint broken
**Detected by:** wp-launch-monitoring scheduled task

## Summary

`GET https://api.world-pulse.io/api/v1/public/signals` is returning **HTTP 400** for every variation of query params tried. The server is up (health check OK), but this specific route is broken by a contradictory validation schema — two validators disagree on the type of the `offset` query parameter.

The rest of the surface is healthy:

- `https://world-pulse.io` → 200 OK
- `https://api.world-pulse.io/health` → 200 OK `{"status":"ok"}`
- GitHub repo reachable, 2 stars

## Evidence

```
$ curl -w "HTTP:%{http_code}" "https://api.world-pulse.io/api/v1/public/signals?limit=1"
HTTP:400
{"success":false,"code":"VALIDATION_ERROR","error":"offset: Expected string, received number"}

$ curl -w "HTTP:%{http_code}" "https://api.world-pulse.io/api/v1/public/signals?limit=1&offset=0"
HTTP:400
{"success":false,"code":"VALIDATION_ERROR","error":"offset: Expected string, received number"}

$ curl -w "HTTP:%{http_code}" 'https://api.world-pulse.io/api/v1/public/signals?limit=1&offset="0"'
HTTP:400
{"statusCode":400,"code":"FST_ERR_VALIDATION","error":"Bad Request","message":"querystring/offset must be number"}

$ curl -w "HTTP:%{http_code}" "https://api.world-pulse.io/api/v1/public/signals"
HTTP:400
{"success":false,"code":"VALIDATION_ERROR","error":"offset: Expected string, received number"}
```

## Root cause signal

Two layers of validation are fighting:

1. One validator (app-level Zod/Joi-style, returns `code: VALIDATION_ERROR`) expects `offset` to be a **string**.
2. Fastify's own schema validator (`FST_ERR_VALIDATION`) expects `offset` to be a **number**.

Whichever request you send, one of them fails. The endpoint is effectively 100% unreachable for public clients.

## Blast radius

This is the main public endpoint backing the live signal stream on `world-pulse.io`. The homepage currently renders "Loading live headlines…" / "Connecting to signal feed…" — consistent with the frontend hitting this endpoint and getting a 400, never resolving. On launch day, every visitor will see a non-loading feed.

## Suggested fix

- Pick one type for `offset` across both layers. Most Fastify + Zod setups coerce query params — the app-level schema should use `z.coerce.number()` (or equivalent) for `offset` and `limit`, so the route validator and Fastify agree on `number`.
- Redeploy API only (`docker compose -f docker-compose.prod.yml up -d --force-recreate api`) and verify:
  ```
  curl -s "https://api.world-pulse.io/api/v1/public/signals?limit=1" | jq '.data | length, .total'
  ```
  should return 200 with a count.
