## Likely causes to investigate (priority order)

1. **wp_api container crash-loop** — per project memory, `pnpm build || true` ships with ~100 TS errors; a runtime throw in a newly-hit codepath can drop the process. Check `docker compose -f docker-compose.prod.yml logs wp_api --tail 200`.
2. **DB connection** — confirm wp_user / worldpulse_db creds still valid; `docker compose -f docker-compose.prod.yml exec wp_api node -e "require('pg')…"` or check `.env.prod`.
3. **wp_scraper down** — scraper is single point of failure for ingest/classify/correlate/trending. If it's wedged, /public/signals may be querying an empty or locked table. Check `scraper:health:*` HASHes in Redis (pw `wp_redis_local`).
4. **nginx upstream mis-pointed** — confirm upstreams still reference `wp_api` (container name), not `api` (compose service).
5. **OpenSanctions / external API failures cascading** — per memory, free tier ~60 req/hour; check if a rate-limit throw is unhandled in the `/public/signals` path.
