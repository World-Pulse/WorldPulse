# ALERT: API 500 — schema drift on /api/v1/public/signals

**Time (UTC):** 2026-04-25 16:35
**Endpoint:** https://api.world-pulse.io/api/v1/public/signals?limit=1
**Status:** HTTP 500
**Web:** OK
**Run:** wp-launch-monitoring scheduled task

## Error
```
{"statusCode":500,"code":"42703","error":"Internal Server Error",
 "message":"select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"}
```

## Diagnosis
PostgreSQL error code **42703** = undefined_column. The query selects `source_url` from `signals`, but that column does not exist in prod. Per memory, signals carries `source_ids` (text[]) — there is no scalar `source_url`. This looks like another schema-drift regression on the public signals route, similar to the Apr 15 fix (alert_tier / source_ids cast).

## Impact
- Public signals API down → /v1/public/signals consumers (homepage live feed, embeds, any third-party API user) get 500s.
- This route has been failing across recent monitor runs (13:08 FAIL, 16:15 FAIL, 16:35 FAIL) — not a new outage, but unresolved on launch day +5.

## Suggested fix
- In the signals public route handler, drop `source_url` from the SELECT list (or replace with a join/derivation from `sources` via `source_ids::text[]`).
- Alternatively, alias the first source URL: `(select url from sources where id::text = any(signals.source_ids) limit 1) as source_url`.
- Re-deploy with `docker compose -f docker-compose.prod.yml up -d --force-recreate api` (per reference_restart_vs_recreate memory: `restart` won't pick up env changes, but for code deploy a force-recreate after image rebuild is needed).

## Status
Repeat failure. Recommend escalating to fix queue NOW — signals API is the public face of WorldPulse.
