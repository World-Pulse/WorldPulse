# ALERT — Launch Day+1 Health Check Failure

**Timestamp:** 2026-04-21T13:07:27Z (local 09:06)
**Severity:** HIGH — documented public API endpoint still failing one day after launch
**Status:** RECURRING (30+ consecutive failures over ~40 hours) — and possibly degraded further (500 vs. the prior 400)

## Failure

`GET https://api.world-pulse.io/api/v1/public/signals?limit=1` → **HTTP 500** (per WebFetch).

This is the same endpoint that has failed on every scheduled pulse since 2026-04-19 17:24 local, originally with a 400 `VALIDATION_ERROR: offset: Expected string, received number` (see `ALERT-2026-04-19T223658Z.md`). If the response has shifted from 400 → 500, this is a regression beyond the known zod validation bug and should be re-diagnosed.

## What IS working

- `GET /api/v1/health` → **200** `{"status":"ok","version":"0.1.0","uptime_s":3246,"map_signals_with_geo":8897,...}` — API process alive, ~54 min uptime (recent restart).
- `GET /api/v1/slop/stats` → **200**, `total_signals_analyzed = 70,336` (up from 64,064 at 15:35 yesterday → ~+6.3K overnight, ingest pipeline healthy).
- `https://world-pulse.io` → **200**, page renders (title "WorldPulse — Global Intelligence Network").
- GitHub repo reachable; star count **3** (+1 overnight).

## Impact

Same as prior alerts — the *documented* public API URL that external consumers and the marketing site's "Try it" snippet land on returns an error. Since the repo is now public and HN/PH traffic continues, every new visitor who copies the docs curl example hits a failure. The fact that `/api/v1/slop/stats` and `/api/v1/signals` work while `/api/v1/public/signals` doesn't makes it worse — a casual observer will assume the platform is broken.

## Root cause (unchanged from 2026-04-19)

Public signals route's `offset` zod schema declares `z.string()`, but upstream (Fastify/middleware) coerces the query param to a number before validation. Fix: `z.coerce.string()`, or `z.union([z.string(), z.number()])` with downstream normalize, or a `"0"` default with numeric-coercion disabled on this route.

**New wrinkle:** the observed status has shifted from 400 → 500. Check recent deploys and logs — a handler that used to reject cleanly may now be throwing an unhandled exception. A 500 suggests the validation error path itself may have regressed.

## Recommended actions

1. **Patch `/api/v1/public/signals` today** — this has been failing for 40+ hours through launch day, not acceptable for the documented public endpoint.
2. Confirm whether the status is now truly 500 (curl it directly; don't rely on WebFetch's summary).
3. After deploy: `curl "https://api.world-pulse.io/api/v1/public/signals?limit=1"` and verify 200 + `data.items[]`.
4. Retire or silence the 30-min scheduled pulse until the route is patched — we're generating alert spam at this cadence with no new information.
