# ALERT — Launch Day Health Check Failure (regression: 400 → 500)

**Timestamp:** 2026-04-20T18:36:47Z (local 14:36)
**Severity:** HIGH — public API endpoint broken on launch day
**Status:** REGRESSED — failure mode changed from 400 VALIDATION_ERROR to 500 SQL ERROR
**Prior alerts:** 19+ failed health pulses since 2026-04-19T21:25Z

## Failure

`GET https://api.world-pulse.io/api/v1/public/signals?limit=1` returns **HTTP 500**.

Response body:
```json
{
  "statusCode": 500,
  "code": "42703",
  "error": "Internal Server Error",
  "message": "select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"
}
```

Postgres error code `42703` = undefined_column. The public-signals route's query
projects `source_url` from the `signals` table, but that column does not exist
in the current prod schema. The column is likely either on `sources` (joined via
`signals.source_ids` text[]) or was renamed.

## What changed since the last alert (2026-04-19T22:36Z)

Previous failure was HTTP 400 with `VALIDATION_ERROR: offset: Expected string,
received number` — a zod/fastify query-param coercion bug. That specific error
is now gone, meaning a patch was shipped. But the patched route has a second bug
in the SQL projection — it references `source_url` on the `signals` table where
that column doesn't exist. Net effect: the endpoint went from 400 to 500,
still not serving traffic. This is a classic "fix one bug, ship another"
regression and it's been live for the full launch-day visibility window.

## What IS working

- `GET /api/v1/health` → 200, uptime 24,658s (~6.8h), map_signals_with_geo=9030.
- `GET /api/v1/signals?limit=1` → 200 with `data.items[]`, real signal payload
  (example: "Green forest fire notification in Laos", Laos, reliability 0.871).
  **The non-public route works.**
- `GET /api/v1/slop/stats` → 200, **total_signals_analyzed = 63,657**
  (up from 55,913 at 18:36 last night — scraper pipeline is still ingesting).
- `https://world-pulse.io` → 200.
- GitHub repo reachable; star count **2**.

## Impact

Same as the prior alert, now worse: anyone hitting the documented public API
(marketing site "Try it" snippet, HN/ProductHunt landings, third-party
integrators) sees an Internal Server Error on launch day. A 500 looks more
broken than a 400 — it reads as "the service is down" rather than "bad
request."

## Root cause (suspected)

Diff the SQL in the public vs. non-public signals route. The non-public route
at `/api/v1/signals` returns full signal objects with `sourceCount` and a
`location` object — it does NOT project `source_url` directly. The public
route is almost certainly trying to project `source_url` as a convenience
field for external consumers, but signals.source_url was never a real column
— source URLs live on the joined `sources` table (see memory:
project_signal_detail_500.md — `signals.source_ids is text[] in prod; queries
joining sources.id must cast both sides to ::text`).

## Recommended immediate actions

1. **Patch the public signals route SQL.** Either drop `source_url` from the
   select list and document that consumers should call `/api/v1/signals/:id`
   for URL detail, OR join `sources` with the `::text` cast and project
   `sources.url as source_url`.
2. After deploy, re-run:
   ```
   curl -s -w "HTTP:%{http_code}\n" "https://api.world-pulse.io/api/v1/public/signals?limit=1"
   ```
   and confirm 200 + `data.items[]` before the next scheduled health pulse.
3. Add a route-level integration test that hits the public signals route with
   no params and with `?limit=1&offset=0`. Both previous bugs would have been
   caught by a single passing test.
4. Consider adding a brain-agent rule: if the same endpoint fails 3 consecutive
   pulses with the same error code, escalate (Slack/email) rather than just
   logging another alert file.
