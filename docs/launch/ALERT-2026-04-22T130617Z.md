## Diagnosis (persistent)
The public signals route selects `source_url` from the `signals` table, but the prod schema stores source references in `source_ids` (text[]) joined to `sources` — there is no `source_url` column on `signals`. This is schema drift between the API build and the live DB.

This is a **persistent regression**, not a transient blip: every health pulse since ~Apr 20 17:00 local has failed with this exact error. See earlier ALERT-*.md files for the pattern.

## Recommended fix (unchanged from prior alerts)
Edit the public signals handler in `apps/api/src/routes/public/signals.ts` (or equivalent):
1. Drop `source_url` from the direct `signals` SELECT.
2. Join/aggregate `sources.url` via `signals.source_ids::text[]` → `sources.id::text` (see `reference_prod_db_creds.md` / `project_signal_detail_500.md` for the cast pattern).
3. Alias as `source_url` in the response so the frontend shape is preserved.
4. Rebuild API image `--no-cache`, redeploy via `deploy-bg.ps1`, and verify `curl -s api.world-pulse.io/api/v1/public/signals?limit=1 | jq .` returns 200.
