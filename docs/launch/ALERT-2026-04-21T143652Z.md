**Severity:** HIGH — launch-day public endpoint down
**Status:** Ongoing (not a new incident — consistent failure since at least 10:06 on 04-20)

## Symptoms
- `GET https://api.world-pulse.io/api/v1/public/signals?limit=1` → HTTP 500
- `GET https://api.world-pulse.io/api/v1/public/signals` → HTTP 500
- `GET https://api.world-pulse.io/health` → 200 OK, `{"status":"ok","timestamp":"2026-04-21T18:36:43.255Z"}`
- `GET https://world-pulse.io` → 200 OK (marketing/frontend renders fine)
- GitHub stars: 3 (no change)

## Diagnosis
- API server is alive (health endpoint responds) — this is a route-level failure, not infra.
- /health responds but /signals 500s → problem is in the signals controller or its DB query path, not nginx, not the container, not the host.
- The health log shows **every 30-minute check since ~10:06 on Apr 20 has reported API FAIL / Signals N/A**. That's ~28+ hours of a broken public endpoint on/after launch day. This is not a transient blip.

## Suspected root causes (in priority order)
1. **Signals controller query failure** — most likely a SQL error on the signals listing query. Candidates given recent memory:
   - `signals.source_ids` is `text[]`; a join against `sources.id` without `::text` cast would 500. See `project_signal_detail_500.md` — the list route may have been missed while the detail route was fixed.
   - A recent column/enum addition (alert_tier, category enum) not synced between code and prod DB.
   - Missing/renamed column referenced by the ORDER BY or WHERE clause.
2. **Scraper/pipeline producing malformed rows** the list endpoint can't serialize (e.g., null on a NOT NULL-assumed field).
3. **Unhandled exception in response serialization** (e.g., a `Date` or `JSON` field with unexpected shape from a new source).

## Immediate next steps for a human operator
1. SSH to prod: `docker logs wp_api --tail 200` — grab the stack trace for the 500. This is the fastest path to root cause.
2. Compare signals list SQL against the fixes in `project_signal_detail_500.md` — apply the `::text` cast if missing.
3. `curl -s https://api.world-pulse.io/api/v1/public/signals?limit=1 -v` from the box to confirm the error body (may contain a hint).
4. If a bad row is suspected, `SELECT id FROM signals ORDER BY created_at DESC LIMIT 1` and try fetching just that id — bisect.
