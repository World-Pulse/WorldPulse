# ALERT — Public API 500 (recurring)

**Time (UTC):** 2026-04-22T15:06:31Z
**Local:** 11:05 ET
**Check:** `GET https://api.world-pulse.io/api/v1/public/signals?limit=1`
**Result:** HTTP 500

## Root cause (from response body)

```
column "source_url" does not exist
```

Query issued by the handler:

```sql
select "id", "title", "category", "severity", "reliability_score",
       "location_name", created_at as published_at, "source_url"
from "signals"
where "status" = $1
order by "created_at" desc
limit $2
```

This is schema drift: the public-signals handler SELECTs `signals.source_url`, but that column does not exist in the prod `signals` table. Signals use `source_ids text[]` (see `reference_api_slop_routes.md`, `project_signal_detail_500.md`). The public endpoint likely needs to either (a) drop `source_url` from its projection and resolve from `source_ids`, or (b) add `source_url` as a denormalized column at ingest time.

## Scope

- Web (`world-pulse.io`) returns 200 — the public site is up, but its "Tracking 0 signals" / "300+ Sources" counter depends on this endpoint and is showing 0.
- Scraper is clearly still running; previous log entries that did get through showed signal counts climbing (55,913 → 64,064 → 70,336 → 73,772 on 2026-04-20/21) despite the public endpoint erroring.
- This failure pattern is present in **every** health-log entry since 2026-04-20 17:24 — roughly 48+ hours of continuous 500s on the primary public read path.

## Recommendation

This is the single most important prod fix outstanding. The site's hero feed and every public reader hitting `/api/v1/public/signals` is broken since launch. Patch the query to match the live schema and ship immediately — no need for a migration, just drop/replace `source_url` in the SELECT list.
