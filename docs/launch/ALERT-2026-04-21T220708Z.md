```json
  "message": "select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"

```

## Root cause

Postgres error code **42703** = `undefined_column`. The public signals query is selecting `source_url` from the `signals` table, but that column does not exist in the prod schema. `signals` stores source references as `source_ids` (text[]) — the canonical source URL lives on joined `sources.url`, not on `signals` directly (see `project_signals_schema_drift.md` and `project_signal_detail_500.md` for prior schema-drift incidents).

This is a code/schema mismatch in the public API route — almost certainly a regression shipped between the 17:24 failure (same symptom) and now. The homepage still renders because it reads via a different path (SSR hits a different query or cached data).

## Likely fix

In the `/api/v1/public/signals` route handler:

1. Remove `source_url` from the SELECT list on `signals`, OR
2. Join `sources` via `signals.source_ids::text[] && array[sources.id::text]` and select `sources.url AS source_url`, OR
3. Rename the column to whatever exists in prod (e.g. the first element of `source_ids` resolved through `sources`, or a denormalized `canonical_url` if one was intended).

Grep the API repo for `source_url` in the public signals query builder — the selected column list is the artifact to fix.
