## Likely suspects (prioritized)
1. **wp_api container** crash-loop or OOM — check `docker logs wp_api --tail 200` and `docker inspect wp_api --format '{{.State.Status}} {{.RestartCount}}'`. Memory notes: MEMORY/feedback_container_started_vs_healthy warns "Started" ≠ healthy.
2. **Postgres reachability from api** — prod creds are `wp_user / worldpulse_db` (MEMORY/reference_prod_db_creds). If a recent redeploy re-read env, a `??` vs `||` regression on a default could null out DSN (MEMORY/feedback_nullish_vs_or_env).
3. **nginx upstream misconfig** — upstreams must reference container names `wp_web` / `wp_api`, not compose service names (MEMORY/project_nginx_upstreams).
4. **scraper/correlate DB writer collisions** — wp_scraper is single point of failure (MEMORY/project_scraper_pipeline); if it's wedging a long transaction on `signals`, the public read path can 500.

## Recommended next actions (for Devon on return)
1. SSH to prod, `docker compose -f docker-compose.prod.yml ps` — get container health for wp_api, wp_web, wp_db, wp_scraper.
2. `docker logs wp_api --tail 500` — look for stack trace, DB connection error, or 500-emitting route.
3. `curl -sS http://localhost:3001/api/v1/public/signals?limit=1` from inside prod box — isolate nginx vs api.
4. If api is unhealthy: `docker compose -f docker-compose.prod.yml up -d --force-recreate api` (restart alone won't re-read env — MEMORY/feedback_restart_vs_recreate).
5. Consider failing over `/api/v1/public/signals` to the seed/fallback pattern already used for `/sanctions/featured` (MEMORY/project_opensanctions_rate_limits) until root cause is fixed — better than 500 on launch day.
