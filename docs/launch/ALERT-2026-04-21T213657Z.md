# ALERT — Launch Day Health Check Failure (REGRESSION)

**Timestamp:** 2026-04-21T21:36:57Z (local 17:36)
**Severity:** HIGH — public API endpoint still down; failure mode has CHANGED
**Status:** NEW FAILURE MODE — prior 400 validation bug appears patched, now returning 500 SQL error

## Failure

`GET https://api.world-pulse.io/api/v1/public/signals?limit=1` returns **HTTP 500**.

Response body:
```json
{
  "statusCode": 500,
  "code": "42703",
  "error": "Internal Server Error",
  "message": "select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"
}
```

Postgres error code `42703` = undefined column. The public signals route is selecting
`source_url` directly from `signals`, but that column doesn't exist on the table in
prod.

## Why this is a regression, not a continuation

Earlier alerts (ALERT-2026-04-19T*) documented the endpoint returning **HTTP 400** with
a zod validation error: `"offset: Expected string, received number"`. That failure mode
is gone. Someone (or a deploy) has since changed the route — the new SQL reflects a
rewrite of the query — and introduced a schema-incompatible column reference. This is
the fourth day in a row the documented public API URL has been broken, now with a new
cause.

## What IS working

- `GET /api/v1/health` → 200 `{"status":"ok","version":"0.1.0","uptime_s":33839,"map_signals_with_geo":9038}` — API process is alive.
- `GET /api/v1/slop/stats` → 200, `total_signals_analyzed = 73,772`.
- `https://world-pulse.io` → 200.
- GitHub repo reachable; **3 stars**.

## Suspected root cause

In prod, signals.source_ids is `text[]` (per project_signal_detail_500 memory), not a
scalar `source_url` column. The public route's SELECT list needs to either:

1. Join `signals.source_ids ::text[]` to `sources.id ::text` and project `sources.url`
   as `source_url` (same cast pattern used elsewhere — see reference note on signal
   detail 500 fix), or
2. Drop `source_url` from the SELECT entirely and let clients dereference sources via
   a separate lookup, or
3. Use a view/computed column that resolves `source_url` server-side from `source_ids`.

## Recommended immediate actions

- Inspect the diff on the public signals route since 2026-04-19. Whatever landed to
  patch the 400 introduced the 500.
- Run `\d signals` on prod DB to confirm column set; compare against whatever the
  route's SELECT claims to project.
- Patch + redeploy. The documented public API URL has now been 400 or 500 for 4
  consecutive days on launch window.
