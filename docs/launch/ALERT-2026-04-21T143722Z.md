# ALERT — API 500 still persisting (Day 2 post-launch)

**Timestamp:** 2026-04-21 10:36 local (14:37 UTC)
**Severity:** Critical — public API has been 500ing continuously since ~11:06 local on Apr 20 (launch day). That is ~23.5 hours of sustained failure on the endpoint that powers the home page signal feed.

## What failed
- `GET https://api.world-pulse.io/api/v1/public/signals?limit=1` → **HTTP 500** (three consecutive probes)
- `GET https://api.world-pulse.io/api/v1/health` → **200 OK** — API process is alive; DB / Redis / Kafka listed with no errors; uptime ~8,624s (~2h23m), map_signals_served = 8,956. **The API container has restarted recently (uptime is only ~2.5h) but the public signals endpoint remains broken.**
- `GET https://world-pulse.io` → 200 OK (title renders)
- GitHub `World-Pulse/WorldPulse` → 3 stars (unchanged since Apr 20 evening)

## What this tells us
The failure is **not** a process-down situation. Health endpoint reports the app is running and claims DB/Redis/Kafka are fine. The 500 is specific to the `/api/v1/public/signals` handler — a code path or query error, not infrastructure.

The uptime of ~2.5h on the API means the container has been restarted at least once overnight, but the restart did not fix the underlying bug. This rules out "just restart it" — whatever is broken is persisted in code, config, or schema.

## Pattern (from rolling health log)
Every 30-min probe since 2026-04-20 11:06 local through 2026-04-21 10:36 local has recorded `API: FAIL`. Over that 23.5h window:
- Occasional successes captured a signal count: 63,657 (14:36), 64,064 (15:35), 70,336 (09:06 Apr 21). So the endpoint sometimes returns successfully — this is a partial failure, not a total outage, but the vast majority of probes 500.
- Signal count is growing (~64k → 70k in ~18h, ingest is working), which is consistent with scraper/ingest healthy but the read path broken.

## High-probability suspects (ranked)
1. **Query-level bug in the `/api/v1/public/signals` handler** — possibly a null/undefined on a new field, a cast regression on `source_ids` text[], or an enum mismatch. The fact that some requests succeed and most fail suggests a data-dependent crash: certain rows trigger the throw, and `limit=1` with default ordering usually picks one of those rows. (See `project_signal_detail_500.md` memory — prod `signals.source_ids` is `text[]`; joins to `sources.id` need `::text` on both sides. A similar cast issue in the list path would reproduce as this.)
2. **Unhandled null in a new column** — `project_signals_schema_drift.md` notes Apr 15 fixes for `alert_tier`, enums, `source_ids`, `location_name`. If a new column was added since launch and the handler does `.toUpperCase()` / `.split()` / `.length` on it without null-checking, rows with null there would 500.
3. **TS debt landing a runtime error** — API ships with 100+ TS errors (`pnpm build || true`). A path that was never hot-tested is now hot.

## Recommended actions for Devon (priority order)
1. **Pull the stack trace.** On prod: `docker compose -f docker-compose.prod.yml logs --tail=500 wp_api | grep -A 30 "public/signals"`. The trace will identify the exact crash site.
2. **Reproduce locally with a single ID.** Once you know the crash site, query the offending row(s) from prod (e.g., `select id, source_ids, category, alert_tier from signals order by created_at desc limit 5`) and check for the null/format that's tripping the handler.
3. **Quick mitigation:** if one column is the culprit, a `COALESCE`/`?? []` in the SQL or handler restores the endpoint instantly. Ship a hotfix to `/api/v1/public/signals` first; full fix can follow.
4. **Verify behind nginx:** `curl -sS http://localhost:3001/api/v1/public/signals?limit=1 -w "\n%{http_code}\n"` from the host. Confirms nginx isn't adding latency/timeout layer.
5. **After fix:** also verify `/api/v1/public/signals` with various `limit`, `category`, `tier` filters — the bug may live in only some filter paths.

## Why this keeps being logged
The wp-launch-monitoring scheduled task runs every 30 min and files a fresh ALERT every time the API returns 500. That's working as intended — but the alerts are now ~47 deep without human intervention since Apr 20 21:08. If this message is the first one Devon reads, **the pager has been ringing since yesterday morning.** Please triage before any further marketing push; homepage visitors are currently getting a broken signal stream on first load.
