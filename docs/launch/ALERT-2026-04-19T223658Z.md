# ALERT — Launch Day Health Check Failure

**Timestamp:** 2026-04-19T22:36:58Z (local 18:36)
**Severity:** HIGH — public API endpoint returning 400 on launch day
**Status:** RECURRING — same failure observed at 17:24, 17:35, 18:06, now 18:36

## Failure

`GET https://api.world-pulse.io/api/v1/public/signals?limit=1` returns **HTTP 400**.

Response body:
```json
{"success":false,"code":"VALIDATION_ERROR","error":"offset: Expected string, received number"}
```

The error persists when `offset=0` is supplied explicitly as a query param — the server
appears to coerce the query string into a number before it reaches a zod schema that
expects `z.string()`. This is a validation/schema bug in the public signals route, not
a platform outage.

## What IS working

- `GET /api/v1/health` → 200 `{"status":"ok"}` — API process is alive.
- `GET /api/v1/signals?limit=1` → 200 with `data.items[...]`, `cursor`, `hasMore`.
  (The non-public route works — no `offset` coercion mismatch.)
- `GET /api/v1/slop/stats` → 200; `total_signals_analyzed = 55,913`.
- `GET /metrics` → 200 (Prometheus format).
- `https://world-pulse.io` → 200, page title "WorldPulse — Global Intelligence Network".
- GitHub repo reachable; star count **2**.

## Impact

Any external consumer (including our own marketing site, third-party integrators, and
the docs "Try it" snippet) that hits the documented `/api/v1/public/signals` route with
default params will receive a 400. On launch day this is a visible, reputation-level
bug. Anyone landing from HN/ProductHunt and trying the public API first will hit it.

## Suspected root cause

The `offset` zod schema on the public signals route declares `z.string()` (probably to
preserve string-cursor semantics), but Fastify or a middleware ahead of it is coercing
the query param to a number. Two likely fixes:

1. Change schema to `z.coerce.string()` or accept `z.union([z.string(), z.number()])`
   and normalize downstream.
2. Give `offset` a default of `"0"` and disable upstream numeric coercion for this
   route's querystring.

Compare against the internal `/api/v1/signals` route — whatever diverges in validation
or querystring parsing is almost certainly the trigger.

## Recommended immediate actions

- After deploy: re-run `curl "https://api.world-pulse.io/api/v1/public/signals?limit=1"`
