## 🚨 Critical issue — `/api/v1/public/signals` 500 regression

**Request:** `GET https://api.world-pulse.io/api/v1/public/signals?limit=5`
**Response:** HTTP 500, reproducible.

### Trend (important)

| Timestamp | Status on `/public/signals` | Interpretation |
|---|---|---|
| 2026-04-19 21:25 UTC (last night precheck) | **400** | Fastify validation / schema rejection — route registered but rejecting input. |
| 2026-04-20 12:01 UTC (now) | **500** | Route reaching handler and throwing. Application-level exception. |

A 400 → 500 transition means someone (or a deploy) changed the route between last night and this morning. The route now reaches the handler but the handler itself errors out. This smells like either (a) a schema-validation bypass that exposed a query bug, or (b) a change to the DB query that hit a known schema-drift issue.

### Why it's a launch blocker

- `/developers` page explicitly lists `https://api.world-pulse.io/api/v1/public/signals` as the primary public endpoint.
- Repo is already public on GitHub (205 commits, 2 stars). Any external developer testing today's announce will follow the docs and immediately see a 500 — worst-possible first impression for an open-source API launch.
- `/health` is 200, so this is NOT a downed container. It's an application error in one route, which means it's fixable in code without infra churn.

### Suggested triage path (fast, <30 min)

1. `ssh` to prod → `docker logs wp_api --tail 200` — grab the 500 stack trace. Curl it first to guarantee a fresh log line: `curl -v https://api.world-pulse.io/api/v1/public/signals?limit=5`
2. Compare the `/public/signals` route handler to the working `/api/v1/signals` handler. They diverged between last night (400 = schema rejection) and now (500 = handler throw), so whatever changed is recent.
3. Probable culprit per project memory: `signals.source_ids` is `text[]` in prod and requires `::text` casts when joining `sources.id` (see `project_signal_detail_500.md`). If the handler does such a join, that's where to look first.
4. Fix in code → redeploy via `deploy-bg.ps1` (API-first, wait for healthy before re-running; `deploy-bg.ps1` back-to-back kills the first build).
5. Re-run this precheck (or just `curl` the two endpoints) to confirm 200.

### Fallback if triage runs past 8:45 AM ET

- Update `/developers` page + README example curls to use `/api/v1/signals` (which works) instead of `/api/v1/public/signals`.
- Or register `/api/v1/public/signals` as a thin alias to `/api/v1/signals` in Fastify and redeploy — a 2-line change.
