**Timestamp (UTC):** 2026-04-25 16:14
**Severity:** Critical (signals API down — core surface broken)
**Run:** scheduled health pulse (wp-launch-monitoring)

## What failed
- `GET https://api.world-pulse.io/api/v1/public/signals?limit=1` → **HTTP 500**
- `https://world-pulse.io` → HTTP 200 (web shell up)
- GitHub stars (World-Pulse/WorldPulse): 3

## Error body
```
{"statusCode":500,"code":"42703","error":"Internal Server Error",
 "message":"select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\",
            \"location_name\", created_at as published_at, \"source_url\"
            from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2
            - column \"source_url\" does not exist"}
```

## Diagnosis
Postgres error code 42703 = undefined column. The public signals query selects `source_url`, but that column does not exist on the `signals` table in prod. This is **schema drift** — code expects a column the DB doesn't have. Likely root causes (in order of likelihood):

1. Recent code change added `source_url` to the select clause but the migration was never applied to prod.
2. Prod DB was restored / recreated from an older snapshot (recall WAL recovery on Apr 3-5).
3. The column lives elsewhere (e.g. `sources.source_url` joined via `signals.source_ids`) and the route is missing the join.

The actual signal URL on this schema lives in `sources.url`, joined via `signals.source_ids` (text[] — must cast to ::text per project_signal_detail_500 memory).

## Pattern
This is **not a one-off** — recent log entries show repeated `API: FAIL` with the same fingerprint (Apr 19 evening, Apr 20 12:53 / 13:08, and now Apr 25). The public signals endpoint has been down across multiple scheduled runs. Web is up so users see a shell with no data.

## Recommended fix
On prod API container:
1. Confirm column missing: `\d signals` in psql — check `source_url` is absent.
2. Either:
   - **Fast fix:** patch the API route to drop `source_url` from the select and source the URL via `sources.url` joined on `source_ids::text[]` (lookup by `sources.id::text`), OR
   - **Schema fix:** add the column via migration + backfill from `sources.url` for the latest source per signal.
3. Redeploy with `--force-recreate` (per feedback_restart_vs_recreate); `restart` won't reload code.
4. Verify: `curl https://api.world-pulse.io/api/v1/public/signals?limit=1` returns 200 + a non-empty array.
