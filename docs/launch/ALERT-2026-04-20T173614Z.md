## Endpoint

`GET https://api.world-pulse.io/api/v1/public/signals?limit=1`

## Response

HTTP 500 with body:

```json
{
  "statusCode": 500,
  "code": "42703",
  "error": "Internal Server Error",
  "message": "select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"
}
```

## Root cause (schema drift)

Postgres error code `42703` = undefined column. The query references `signals.source_url`, which does not exist in the production `signals` table. Per memory (`reference_api_slop_routes.md`, `project_signals_schema_drift.md`), signals join to sources via `signals.source_ids text[]` → `sources.id`; there is no direct `source_url` column on `signals`.

This is the same class of bug as the Apr 15 schema drift fixes — a public endpoint handler is out of sync with the prod schema.

## Likely fix

In the `/api/v1/public/signals` handler, replace the `source_url` selection with either:
- a join: `LEFT JOIN LATERAL (SELECT url FROM sources WHERE id::text = ANY(signals.source_ids)) s ON true` and select `s.url AS source_url`, or
- remove `source_url` from the select list if the public feed doesn't need it.

Also worth grep'ing the API for any other `signals.source_url` references and adding a smoke test against `/api/v1/public/signals?limit=1` in the deploy pipeline so this can't regress silently.
