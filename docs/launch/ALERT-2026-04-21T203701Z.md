## NEW — concrete SQL error captured

Today's 500 response exposes the specific failure mode, which does not appear to have been recorded in prior alerts:

```
{"statusCode":500,"code":"42703","error":"Internal Server Error",
 "message":"select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\",
  \"location_name\", created_at as published_at, \"source_url\" from \"signals\"
  where \"status\" = $1 order by \"created_at\" desc limit $2
  - column \"source_url\" does not exist"}
```

PostgreSQL error code `42703` = undefined_column. The API's `/public/signals` handler is selecting `source_url` from `signals`, but that column does not exist in the prod schema. This is schema drift — same class of issue as `project_signals_schema_drift.md` (resolved Apr 15 for alert_tier/enums/source_ids/location_name) but on a column that was evidently missed or re-introduced.

Possible causes, ranked:
1. **Code regression since Apr 15** — a handler was updated to read `source_url` without a matching migration. Check recent commits touching `api/src/routes/public/signals*` or any query builder with `signals.source_url`.
2. **Migration rolled back / never applied on prod** — an earlier migration added `source_url` to `signals` but was not run on this DB. Prod `signals.source_url` may live in `sources` table instead (signals join sources via source_ids text[] cast — see `project_signal_detail_500.md`).
3. **TS build silently shipped broken code** — API Dockerfile uses `pnpm build || true` and ships with 100+ TS errors (`project_api_typescript_debt.md`), so a type-unsafe column reference would not block deploy.

## Recommended next actions

1. On prod, inspect the actual columns:
   ```
   docker exec wp_postgres psql -U wp_user -d worldpulse_db -c "\d signals" | grep -i source
   ```
   If `source_url` truly doesn't exist on `signals`, the query in the public/signals route is wrong.
2. Grep the API source for `source_url` references on the signals table:
   ```
   grep -rn "source_url" api/src/ | grep -v node_modules
   ```
   Confirm whether the intent is `signals.source_url` (needs migration) or `sources.url` (needs a join — similar pattern to the signal-detail 500 fix).
3. Quick fix options, in order of safety:
   - **Patch the query** to alias a computed URL from the `sources` join (preferred; matches existing schema).
   - **Add migration** `ALTER TABLE signals ADD COLUMN source_url text` + backfill — only if product intent is to denormalize.
4. After patch: redeploy with `--no-cache` per `project_apr15_prod_fixes.md`, and force-recreate per `feedback_restart_vs_recreate.md`:
   ```
   docker compose -f docker-compose.prod.yml build --no-cache api
   docker compose -f docker-compose.prod.yml up -d --force-recreate wp_api
   ```
5. Smoke test: `curl -s https://api.world-pulse.io/api/v1/public/signals?limit=1 | jq .`
