## Context
- This is a recurring failure pattern — the health log shows API: FAIL across virtually every check since 2026-04-20 17:06, with only one successful read (17:36 returning 73,772 signals).
- The marketing site stays up while the API is down, suggesting the failure is isolated to the `wp_api` container or its upstream dependencies (DB / scraper-derived materialized views), not the nginx/web tier.
- Two days post-launch with the public API returning 500s means every "live" surface that reads from `/api/v1/public/*` is broken for visitors.

## Likely culprits (per memory)
1. **wp_api crash-loop** — `project_api_typescript_debt.md` notes the API ships with `pnpm build || true` and 100+ TS errors; runtime failures are plausible.
2. **Scraper down** — `project_scraper_pipeline.md`: if `wp_scraper` is dead, downstream surfaces empty within 24h; signals query may be hitting a broken view.
3. **Schema drift / signals.source_ids cast** — `project_signal_detail_500.md` documents text[] cast issues that previously caused 500s on signals routes.
4. **Container started ≠ healthy** — `feedback_container_started_vs_healthy.md`: a "Started" wp_api can still be crash-looping; need `docker inspect` + `curl localhost:3001` + `docker logs --tail=100 wp_api` to confirm.

## Recommended next actions (for Devon)
1. SSH to prod, run: `docker compose -f docker-compose.prod.yml ps` and `docker logs --tail=200 wp_api`
2. Hit the API directly inside the network: `docker exec wp_api curl -sS http://localhost:3001/api/v1/public/signals?limit=1`
3. Check `wp_scraper` health: `docker logs --tail=100 wp_scraper` and Redis keys `scraper:health:*` / `scraper:stability:*`
4. If logs show TS runtime errors, redeploy with `--no-cache --force-recreate` (per `feedback_restart_vs_recreate.md`)
5. If the failure is scraper-driven (no fresh signals), the API may be choking on a stale/corrupt query — consider rolling back the most recent signals migration

## Pattern note
The health log's near-100% FAIL rate over 48+ hours suggests this monitoring task may be reporting on a known-broken endpoint. If the API has been intentionally returning 500 since launch, this scheduled task should either (a) be paused, or (b) be pointed at a different healthcheck endpoint (e.g. `/health` or `/api/v1/health`) that actually reflects API liveness rather than a route that depends on potentially-broken downstream data.
