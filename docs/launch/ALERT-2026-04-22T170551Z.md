## Likely root causes to check (in order)

2. **nginx upstream** — `nginx.conf` must point at `wp_api` (container name), not `api` (service name). Regression here would 502/500 everything.
3. **Postgres connectivity** — creds are `wp_user` / `worldpulse_db`. A wrong `DATABASE_URL` in `.env.prod` would surface as 500s on every data route.
4. **Schema drift / migration** — recent changes to `signals.source_ids` (text[]) or enum casts; an unran migration on prod would throw 500.
5. **wp_scraper** — if scraper is dead, ingest stops but `/public/signals` should still read existing rows. If signals endpoint is 500-ing, this is NOT a scraper-only issue.
