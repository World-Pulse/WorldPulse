## API error body

```json
{
  "statusCode": 500,

  "error": "Internal Server Error",
  "message": "select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"

```

## Diagnosis

Postgres error code `42703` = undefined column. The `/api/v1/public/signals` route handler is selecting `source_url` directly from `signals`, but in the current production schema that column does not exist on `signals` — signal-to-source linkage is stored in `signals.source_ids` (`text[]`) and the canonical URL lives on the `sources` table (see memory `project_signal_detail_500.md`, which already flagged casting issues around this join).

This is the same failure pattern that has been recorded in the health log every 30 min since at least 2026-04-20 17:24 UTC — API has been returning 500 for ~26 hours of launch + post-launch monitoring. Website is up, but the public signals API (and by extension anything that feeds from it) is broken.

## Fix direction (for the next on-call pass)

1. In the API route building that SELECT, drop the bare `source_url` column — it doesn't live on `signals`. Either:
   - join `sources` via `signals.source_ids::text[]` and select the first matching `sources.url`, casting both sides to `::text`, **or**
   - return `source_ids` and resolve URLs client-side.
2. Redeploy via `deploy-bg.ps1` (single run — see `feedback_deploy_bg_concurrency.md`) and force-recreate the api container so env vars re-read (`feedback_restart_vs_recreate.md`).
3. Verify with `curl -s -o /dev/null -w "%{http_code}" https://api.world-pulse.io/api/v1/public/signals?limit=1` returns 200 before closing the alert.
