## Context

API has been returning 500 across every scheduled health pulse run today and since late 2026-04-19 (see prior health log entries at 17:24, 17:35, 18:06, 18:36, 19:07, 19:35, 20:05, 20:35, 09:05, 09:35). A standing alert already exists at `ALERT-2026-04-19T212516Z.md` — this is not a new incident, it is the same outage still unresolved at T+~17h.

Signal count could not be extracted because the endpoint itself is the failure. Earlier snapshots (~56,000 signals as of 19:35 yesterday) came from a period when the endpoint was transiently returning cached data; current probe returns 500 with no body parseable by the fetcher.

## Recommended next actions

1. SSH into prod and check `docker ps` + `docker logs wp_api --tail 200` — is the container crash-looping? See `feedback_container_started_vs_healthy.md`.
2. Check `docker logs wp_scraper --tail 100` — scraper is upstream of signals and is a single point of failure (see `project_scraper_pipeline.md`).
3. If restart needed, use `docker compose -f docker-compose.prod.yml up -d --force-recreate wp_api` (plain `restart` will not re-read env — see `feedback_restart_vs_recreate.md`).
4. Verify nginx upstream still points at `wp_api` (container name), not `api` (compose service) — see `project_nginx_upstreams.md`.
5. If a fresh code issue, this may be related to the ongoing API TypeScript debt (`project_api_typescript_debt.md`) — `pnpm build || true` means a broken build still ships.
