# ALERT: API 500 — schema drift on /api/v1/public/signals

**Detected:** 2026-04-25 17:05 UTC (scheduled health pulse)
**Severity:** P0 — public API endpoint returning 500 on launch-day monitoring path
**Recurrence:** This same failure has appeared in every scheduled health pulse since at least 2026-04-19. Web (https://world-pulse.io) is OK; only the API surface is broken.

## Failure

`GET https://api.world-pulse.io/api/v1/public/signals?limit=1` → HTTP 500

```json
{
  "statusCode": 500,
  "code": "42703",
  "error": "Internal Server Error",
  "message": "select \"id\", \"title\", \"category\", \"severity\", \"reliability_score\", \"location_name\", created_at as published_at, \"source_url\" from \"signals\" where \"status\" = $1 order by \"created_at\" desc limit $2 - column \"source_url\" does not exist"
}
```

## Diagnosis

Postgres error code **42703** = `undefined_column`. The query selects `source_url` from `signals`, but that column does not exist in the production `signals` table. Per memory (`reference_api_slop_routes.md`, `project_signals_schema_drift.md`, `project_signal_detail_500.md`), `signals.source_ids` is a `text[]` and the canonical pattern is to dereference source URL via a join on `sources` (with `::text` casting on both sides) — there is no `signals.source_url` scalar column.

This is **schema drift**: the API code was written against a schema that has `signals.source_url`, but production migrated away from that shape. Either the column was dropped/renamed, or it was never deployed.

## Likely fix paths (pick one)

1. **Add the column back** — `ALTER TABLE signals ADD COLUMN source_url TEXT;` and backfill from `sources` via `source_ids[1]`. Fastest unblock, but treats the symptom.
2. **Fix the query** — change `apps/api/src/routes/public/signals.ts` (or wherever the public list query is built) to project `source_url` from a join: `LEFT JOIN sources s ON s.id::text = ANY(signals.source_ids::text[])` and select `s.url AS source_url`. Correct fix, requires a small API redeploy.
3. **Drop `source_url` from the projection** if the public payload doesn't need it — the cheapest patch.

Recommend option 2: it matches the canonical pattern already used by `/api/v1/signals/:id`.

## Impact

- Any client calling `/api/v1/public/signals` (homepage live-feed component, public widget, third-party API consumers) is getting 500s.
- Web shell loads (200), but signal data won't populate from this endpoint.
- This has been broken for **6+ days** based on the health log — worth checking whether the dashboard surfaces a different path that masks this from manual eyeballing.

## Recommended immediate actions

1. SSH to prod and confirm: `\d signals` in psql — verify `source_url` is genuinely absent.
2. Patch the query (option 2 above), redeploy API with `--no-cache`, verify with the same `curl`.
3. Add a synthetic check in CI for this exact endpoint so the next schema drift fails the deploy instead of silently 500-ing for a week.
