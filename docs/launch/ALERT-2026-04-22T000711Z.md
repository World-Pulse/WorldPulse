## Failure detail

The public signals API returns HTTP 500 with this Postgres error:

```
{"statusCode":500,"code":"42703","error":"Internal Server Error",
 "message":"select \"id\", \"title\", \"category\", \"severity\",
 \"reliability_score\", \"location_name\", created_at as published_at,
 \"source_url\" from \"signals\" where \"status\" = $1
 order by \"created_at\" desc limit $2
 - column \"source_url\" does not exist"}
```

Postgres error code **42703** = undefined column. The query selects `source_url` from `signals`, but that column does not exist in the prod schema.

## Recommended fix

The `signals` table does not have `source_url` — per memory `reference_api_slop_routes.md` and `project_signal_detail_500.md`, signals link to sources via `source_ids text[]` joined to `sources.url`. Two options:

1. **Hot fix the query** (preferred — fastest path to GREEN): change the public signals route handler to drop `source_url` from the select and instead resolve it via a join on `sources` using `source_ids[1]::text = sources.id::text` (note the text-cast on both sides — see `project_signal_detail_500.md`).
2. **Add column + backfill**: if `source_url` is meant to be denormalized onto `signals` for performance, add the column, backfill from `sources`, and update the scraper to populate on insert. Slower but cleaner.

Hot fix recommended for launch-day stability; backfill can ship in the post-launch sprint.
