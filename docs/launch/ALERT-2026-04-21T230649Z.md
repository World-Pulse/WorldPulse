## Diagnosis

Postgres error code **42703** = `undefined_column`. The API query at `/api/v1/public/signals` is selecting `source_url` from the `signals` table, but that column does not exist in the current prod schema. The signals table uses `source_ids` (text[]) referencing the `sources` table (per prior schema drift fix Apr 15 — see `project_signals_schema_drift.md`).

This is a regression or a query-side bug in the public signals route — the column reference in the SQL is wrong.

## Impact

- Public signals API endpoint is fully broken (500 on any call).
- Signal count cannot be reported to monitoring.
- Live feed / public consumers cannot retrieve signals.
- Web shell still renders (likely cached / client-side fallbacks), but any data-fetching from this endpoint will fail.

## Recurrence

This same error has been logged on nearly every pulse throughout launch day and into 2026-04-21 (see the `ALERT-2026-04-20T*` and `ALERT-2026-04-21T*` series in this folder). **This is an unresolved, persistent outage, not a transient blip.** It needs human intervention — either:

1. Fix the SQL in the public signals route to drop the `source_url` column and use `source_ids` + a join against `sources`, **or**
2. Add a `source_url` column to the `signals` table (likely as a denormalized convenience field) and backfill it.

## Recommended action

Option (1) is safer and more correct given the Apr 15 schema consolidation. Grep the API for `source_url` in any signals-related route/query and replace with the correct projection. Likely file: `apps/api/src/routes/public/signals.ts` (or equivalent).
