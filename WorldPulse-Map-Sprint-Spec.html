Viral Map Features Tier 1 — Launch Differentiators
1. Animated Wind Particle Flow
SHIPPED
earth.nullschool.net went viral purely because of this effect. Millions of animated particles flowing across the globe create an instant "whoa" moment that people screenshot and share. This is the single highest-ROI visual enhancement for WorldPulse's launch.
What It Does

Renders thousands of tiny particles flowing across the map surface, driven by real wind velocity data. Particles trace wind currents in real-time, creating hypnotic animated streamlines. Users can toggle the layer on/off and adjust particle density. On the WorldPulse globe, this transforms a static signal map into a living, breathing intelligence surface.

Technical Architecture
┌──────────────────────────────────────────────────────────────┐
│  Wind Data Pipeline                                          │
│                                                              │
│  GFS/NOAA API  ──→  Fastify Cron (6h)  ──→  Redis Cache     │
│  (GRIB2 JSON)       /api/v1/wind/grid       (wind:grid:*)   │
│                          │                                    │
│                          ▼                                    │
│               UV Grid Texture (256×128)                       │
│               [u_velocity, v_velocity] per cell              │
└──────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────┐
│  WebGL Particle Renderer (Custom MapLibre Layer)             │
│                                                              │
│  1. Particle State Texture (FLOAT32)                         │
│     - 65,536 particles (256×256 texture)                     │
│     - Each pixel = [lng, lat, age, speed]                    │
│                                                              │
│  2. Update Shader (Fragment)                                 │
│     - Sample wind UV at particle position                    │
│     - Advance position: pos += wind_uv * dt * speedFactor   │
│     - Age particles, respawn when age > maxAge               │
│     - Random respawn position for dead particles             │
│                                                              │
│  3. Render Shader (Vertex + Fragment)                        │
│     - Project particle lat/lng → screen coords               │
│     - Trail: render previous N positions as fading line      │
│     - Color by wind speed (blue → cyan → green → yellow)     │
│     - Alpha fade based on age                                │
│                                                              │
│  4. Fade Framebuffer                                         │
│     - Composite previous frame at 97% opacity                │
│     - Creates smooth trail effect without storing history     │
└──────────────────────────────────────────────────────────────┘
Implementation Tasks
  • Wind data endpoint — New route GET /api/v1/wind/grid that fetches GFS 1° grid from NOAA's JSON API, parses U/V wind vectors, caches in Redis for 6 hours. Returns 256×128 flat array of [u, v] pairs.
  • Cron scheduler — Background job in Fastify using @fastify/schedule or node-cron to refresh wind grid every 6 hours (matches GFS update cycle). Warm cache on API startup.
  • WebGL custom layer — Implement maplibregl.CustomLayerInterface with onAdd(), render(), onRemove(). Two-pass rendering: update pass (advance particles via transform feedback or ping-pong textures) + draw pass (render particles as GL_POINTS with trail fade).
  • Particle state management — Initialize 65,536 particles at random positions. Store state in a Float32 texture (ping-pong between two framebuffers). Each frame: read state → sample wind → write new state → render.
  • Wind texture upload — Convert grid array to LUMINANCE_ALPHA WebGL texture (U in luminance, V in alpha). Bilinear interpolation in shader gives smooth wind field between grid cells.
  • Trail rendering — Use a screen-space fade framebuffer. Each frame: draw current particles on top of previous frame faded to 97% opacity. This creates the signature flowing trail effect without storing particle history.
  • UI toggle — Add "Wind" button to the map layer toolbar (alongside existing Satellite, Heatmap toggles). Slider for particle density (16k / 32k / 65k). Default: ON at 32k particles for first-time visitors.
  • Performance guard — Detect GPU via WEBGL_debug_renderer_info. Drop to 16k particles on mobile/integrated GPUs. Disable entirely if WebGL2 unavailable. Target: 60fps on M1 MacBook, 30fps on mid-range Android.

2. Time Slider + Historical Playback
SHIPPED
Ventusky's timeline scrubber is what keeps users playing with the app for minutes. Letting users "rewind" the map to watch intelligence signals unfold over time turns WorldPulse from a dashboard into a storytelling tool. No competitor in the OSINT space has this. "Watch the Ukraine conflict unfold over the last 7 days" — that's a tweet that writes itself.
What It Does

A draggable timeline bar at the bottom of the map. Users scrub through time (1h, 6h, 24h, 7d, 30d ranges) and see signals appear/disappear on the map as they were published. Play button auto-advances at configurable speed. Signals fade in when they enter the time window and pulse briefly, creating a visual "ripple" effect across the globe.

Technical Architecture
┌──────────────────────────────────────────────────────────────┐
│  Backend: Historical Signal Bucketing                        │
│                                                              │
│  GET /api/v1/signals/map/timeline                            │
│  ?range=7d&bucket=1h&category=all&bbox=...                   │
│                                                              │
│  PostgreSQL Query:                                           │
│  SELECT                                                      │
│    date_trunc('hour', created_at) AS bucket,                 │
│    json_agg(json_build_object(                               │
│      'id', id, 'lat', ST_Y(location::geometry),              │
│      'lng', ST_X(location::geometry),                        │
│      'severity', severity, 'title', title,                   │
│      'category', category                                    │
│    )) AS signals                                             │
│  FROM signals                                                │
│  WHERE created_at >= NOW() - '7 days'::interval              │
│    AND status IN ('verified','pending')                      │
│  GROUP BY bucket ORDER BY bucket                             │
│                                                              │
│  Response: { buckets: [ { t: ISO, signals: [...] }, ... ] }  │
│  Cache: 60s (Redis)                                          │
└──────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────┐
│  Frontend: Timeline Controller                               │
│                                                              │
│  ┌─────────────────────────────────────────────┐             │
│  │  TimeSlider Component                        │             │
│  │  ├── Range selector (1h/6h/24h/7d/30d)      │             │
│  │  ├── Draggable scrubber thumb                │             │
│  │  ├── Bucket histogram (signal density)       │             │
│  │  ├── Play/Pause + Speed (1x/2x/4x/8x)      │             │
│  │  └── Current timestamp display               │             │
│  └─────────────────────────────────────────────┘             │
│                                                              │
│  Playback Loop:                                              │
│  1. Fetch all buckets for selected range (single request)    │
│  2. On play: advance currentTime by bucket interval          │
│  3. Filter visible signals: created_at <= currentTime        │
│  4. Update GeoJSON source with filtered subset               │
│  5. New signals entering window get 500ms scale-in animation │
│  6. requestAnimationFrame drives smooth scrubbing            │
└──────────────────────────────────────────────────────────────┘
Implementation Tasks
  • Timeline API endpoint — GET /api/v1/signals/map/timeline with params: range (1h|6h|24h|7d|30d), bucket (auto-calculated: 5min for 1h, 15min for 6h, 1h for 24h/7d, 6h for 30d), category, bbox. Uses PostGIS date_trunc aggregation. Cache 60s in Redis.
  • TimeSlider component — New component apps/web/src/components/map/TimeSlider.tsx. Full-width bar fixed at bottom of map. Contains: range tabs, SVG histogram of signal density per bucket, draggable thumb, play/pause/speed controls, formatted timestamp. CSS transitions for smooth scrubbing.
  • Histogram visualization — SVG bar chart above the scrubber showing signal density per bucket. Bars colored by dominant severity in bucket. Gives users a visual preview of where "interesting" time periods are before scrubbing.
  • Playback engine — useTimelinePlayback hook. Manages currentTime, isPlaying, speed (1x/2x/4x/8x). Uses requestAnimationFrame to advance time smoothly. Exposes visibleSignals computed from cached bucket data.
  • Map integration — When timeline is active, replace the live signal source with timeline-filtered signals. Each frame: compute which signals are visible at currentTime, update map.getSource('signals').setData(). Signals entering the window get a 500ms scale-up animation via interpolated circle-radius.
  • Signal entrance animation — Signals that just "appeared" (within last bucket) render with a brief expanding ring + opacity fade-in. Uses the existing new-highlight layer pattern but triggered by timeline position, not WebSocket.
  • URL state sync — Add tl (timeline range) and tt (timeline timestamp) to URL params. Restoring a permalink with timeline params opens the map at that exact moment in time.
  • Keyboard shortcuts — Space = play/pause, Left/Right arrows = step one bucket, Shift+arrows = 10 buckets. Numbers 1-5 = range presets.

3. Shareable Map Permalinks + Social Cards
SHIPPED
earth.nullschool.net's virality depended on shareable URLs. Every interesting map view needs to be one click away from a Twitter post. Without this, all the visual magic stays trapped on the user's screen. This is the distribution mechanism for everything else we're building.
What It Does

Every map state — position, zoom, active layers, time slider position, selected signal — is encoded into the URL hash. A "Share" button copies the permalink and optionally generates a static screenshot preview card for social media (Open Graph). Pasting a WorldPulse map link on Twitter/Reddit/Slack shows a rich preview with the actual map view.

Implementation Tasks
  • URL hash encoder/decoder — Serialize full map state to compact URL: #@lat,lng,z|layers=wind,heat|t=7d:2026-03-28T14:00Z|sig=abc123. Use base62 encoding for coordinates to keep URLs short. Decode on page load to restore exact state.
  • Share button UI — Floating action button (bottom-right cluster). Click → copies permalink to clipboard with toast confirmation. Long-press/right-click → share menu with Twitter, Reddit, Copy Link, QR Code options.
  • Deep link restoration — On map/page.tsx mount, parse hash before initializing MapLibre. Set initial viewport, activate layers, restore timeline position, highlight selected signal. All before first render to avoid visual jumps.
  • OG image generation — Server-side endpoint GET /api/v1/map/og?state=ENCODED_STATE that renders a 1200×630 static map image using MapLibre GL Native (Node.js) or a headless Puppeteer screenshot. Cached in Redis (1h TTL, keyed by state hash).
  • Dynamic meta tags — Next.js generateMetadata() in map route reads URL params and returns OG tags: title ("WorldPulse — Live Intelligence Map"), description (signal count + top category), image (OG endpoint URL).
  • QR code generator — Client-side QR code from permalink URL using qrcode npm package. Displayed in share popover for mobile-to-mobile sharing.
URL Schema
Full URL: https://world-pulse.io/map#@38.9,-77.04,8|l=wind,sig|t=7d:1711641600|s=abc123

Segments:
  @lat,lng,zoom          Position (required)
  l=layer1,layer2        Active layers (wind|heat|sat|adsb|naval|cam|risk)
  t=range:timestamp      Timeline state (optional)
  s=signalId             Highlighted signal (optional)
  b=dark|satellite|terrain  Basemap (optional, default=satellite)

4. Signal Density Heatmaps (Enhanced)
SHIPPED
The existing heatmap mode is basic — it uses MapLibre's built-in heatmap layer. MarineTraffic's vessel density maps are stunning because they use custom gradient coloring, animated glow pulses on hot zones, and zoom-adaptive kernel sizing. Upgrading our heatmap to this level makes the "intelligence hotspot" concept visually undeniable.
What It Does

Replaces the basic heatmap with a multi-layer density visualization. Hot zones pulse with a subtle animated glow. Color gradient shifts by category (red=conflict, blue=climate, green=health). Zoom-adaptive: at low zoom shows continental heat; at high zoom shows street-level clusters. Severity acts as a weight multiplier — critical signals contribute 4x to heat intensity.

Implementation Tasks
  • Weighted heatmap layer — Replace current heatmap with severity-weighted version. Weight mapping: critical=4, high=3, medium=2, low=1, info=0.5. Use MapLibre heatmap-weight property with match expression on severity field.
  • Category-specific color ramps — Different gradient per category filter: conflict (black→red→orange→white), climate (black→blue→cyan→white), health (black→green→lime→white), economy (black→purple→pink→white), all (black→blue→cyan→yellow→white). Apply via heatmap-color interpolation.
  • Zoom-adaptive radius — heatmap-radius interpolated from 15px at zoom 2 → 30px at zoom 6 → 50px at zoom 10 → 80px at zoom 14. Creates smooth visual transition from global overview to local detail. Current version is static.
  • Animated glow pulse — Overlay a second semi-transparent heatmap layer with a pulsing heatmap-opacity (0.3→0.6 sine wave, 3s cycle). Creates the "breathing" hotspot effect. Uses same data source, slightly larger radius.
  • Hotspot labels — At zoom >5, show floating labels on the densest heat zones: "Kyiv Region — 47 signals" with a severity breakdown mini-bar. Uses the existing /api/v1/signals/map/hotspots endpoint. Labels avoid overlap via collision detection.
  • Smooth transitions — When toggling heatmap mode, animate opacity from 0→1 over 400ms. When switching categories, cross-fade between color ramps.
