🔍 Root Cause Diagnosis
@worldpulse/types build took 1,615 seconds. A 2-file TypeScript declaration build should take 3-5 seconds. The server is resource-starved (OOM/swap thrashing) while building 3 services concurrently. SSH session drops (client_loop: send disconnect: Connection reset) KILL the in-flight build because it runs foreground inside the SSH session. Also seeing ECONNRESET from npm registry — network is flaky on the VPS.
✅ Fix: scripts/deploy-bg.ps1 (background deploy)
Uses setsid nohup on server so the build detaches from SSH session. Connection drops no longer kill the build. Progress goes to /var/log/worldpulse-deploy.log.
# Step 1: Deploy API only (fixes all 3 P0s + cascading P1s)
.\scripts\deploy-bg.ps1 -Service api

# Step 2: From ANOTHER terminal, watch progress
.\scripts\deploy-bg.ps1 -Tail

# Step 3: When API is up, verify
curl https://api.world-pulse.io/api/v1/public/signals

# Step 4: Deploy web (only after API confirmed)
.\scripts\deploy-bg.ps1 -Service web

# Check status anytime
.\scripts\deploy-bg.ps1 -Status
⚠ Why API-first (not full deploy)
The API build is fast (~30s for just the TypeScript compile). Web build with Next.js 16 + Turbopack + Sentry takes 20+ minutes and is the primary cause of server thrashing. The 16 API route fixes are the ENTIRE critical path — they unblock map, clusters, signal detail, space weather, briefing, headlines, trending, GDELT. Web container stays on old image until API is healthy.
